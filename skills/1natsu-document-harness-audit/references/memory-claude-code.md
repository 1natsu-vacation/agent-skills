# Claude Code auto-memory リファレンス

メモリ監査（SKILL.md ステップ3）を Claude Code の auto-memory に対して行う時の具体知識。

## 格納場所

```
~/.claude/projects/<プロジェクトパスの変換形>/memory/
```

`<プロジェクトパスの変換形>` は、プロジェクトディレクトリの絶対パスの `/` と `.` を `-` に置換したもの（例: `/Users/x/ghq/github.com/org/repo` → `-Users-x-ghq-github-com-org-repo`）。監査を実行しているエージェント自身がこのプロジェクトで使っているメモリディレクトリがそのまま監査対象になる。

**ユーザー固有・マシン固有・パス固有**である点に注意: クラウド共有されず、同じリポジトリでも clone 位置（パス）が変わると別ディレクトリになる。「メモリはチーム知識にならない」原則の根拠がこの性質。

## 構造

- **1ファイル = 1事実** の markdown。frontmatter を持つ:

  ```markdown
  ---
  name: <kebab-case スラッグ>
  description: <1行サマリー>
  metadata:
    type: user | feedback | project | reference
  ---

  <本文。関連メモリへは [[name]] 形式でリンク>
  ```

- **`MEMORY.md`**: インデックスファイル。各メモリへの1行ポインタ（`- [Title](file.md) — hook`）の羅列。セッション開始時にコンテキストへロードされるのはこのインデックスで、個々のメモリファイルは必要時に読まれる

## type ごとの蒸留判定の目安

判定はあくまで内容で行う（SKILL.md の2軸判定）。type は初期仕分けの目安:

| type | 意味 | 目安 |
|------|------|------|
| `user` | ユーザーが誰か（役割・専門性・好み） | 個人系。原則蒸留対象外（陳腐化検査のみ） |
| `feedback` | 作業の進め方への指導・矯正 | 内容次第。プロジェクトの規約・制約なら蒸留候補、個人の好みなら残す |
| `project` | 進行中の作業・目標・制約 | 恒久的な制約・設計判断なら蒸留候補。進行状態（「PR #5 未マージ」等）は一時メモとして残し、完了していれば削除候補 |
| `reference` | 外部リソースへのポインタ（URL・ダッシュボード・チケット） | チームに有用なら docs へ。個人用ブックマークなら残す |

## 削除・書き換え時の連動

メモリファイルを削除・改名する時は、以下を同時に処理する:

1. `MEMORY.md` の該当行を削除・更新する（インデックスと実体の不整合は、それ自体が次セッションの参照切れになる）
2. 他のメモリ本文から `[[name]]` で参照されていないか確認し、あればそのリンクも処理する
