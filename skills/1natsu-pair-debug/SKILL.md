# Pair Debug（二人三脚デバッグ）

コードの静的分析だけでは原因を特定できない問題を、人間の「目」と「手」を借りて解決する協調デバッグモード。

AIは仮説の立案・検証コードの仕込み・結果の分析を担当し、人間はアプリの実行・画面の確認・ログの報告を担当する。推測で結論に飛びつくのではなく、観測事実に基づいて原因を絞り込む。

## モードの開始

### ユーザーからの明示的な起動

以下のようなフレーズでユーザーがモードを要求した場合、即座にモードに入る：

- 「二人三脚デバッグしよう」「ペアデバッグしよう」「一緒にデバッグして」「pair debug」
- 「一緒に調査しよう」「実際に動かして確認したい」「ログ仕込んで調べたい」

### AIからの提案

以下の状況を検知した場合、モードへの移行を**提案する**（勝手に入らない）：

- コードを読んだが、複数の原因仮説があり静的分析だけでは絞れない
- 環境依存・タイミング依存の問題が疑われる
- 同じ修正を試しても解決しない（堂々巡りの兆候）
- ランタイムの実際の値・状態を見ないと判断できない

提案の例：

> コードを見る限り原因の候補が複数あり、実際のランタイムの挙動を確認しないと絞り込めません。二人三脚デバッグモードに入って、一緒に調査しませんか？

ユーザーが同意したらモードに入る。断られたら通常の自律モードで最善を尽くす。

## デバッグループ

モード中は以下のサイクルを繰り返す。1ステップずつ着実に進め、飛躍しない。

### 1. 仮説を立てる

現時点で考えられる原因候補を列挙する。各仮説に「何が観測できればこの仮説を確認/棄却できるか」を添える。

```
現在の仮説：
A) APIレスポンスのフォーマットが想定と異なる
   → 確認方法: fetchの戻り値をログに出す
B) stateの更新タイミングがレンダリングと競合している
   → 確認方法: useEffect内でstateの値をログに出す
```

### 2. 観測ポイントを仕込む

仮説を検証するための最小限の計装コードを仕込む。

**仕込むもの（例）：**
- `console.log` / `console.table` でのランタイム値の出力
- 一時的なUI表示（変数の値を画面上にレンダリング）
- 条件分岐ごとの通過確認ログ
- try-catchでの詳細エラー出力
- ネットワークリクエスト/レスポンスのダンプ

**仕込む際のルール：**
- デバッグ用コードには `// [PAIR-DEBUG]` コメントをつけ、後から識別・除去しやすくする
- 既存のビジネスロジックは変更しない（観測コードの追加のみ）
- 一度に仕込む観測ポイントは必要最小限に絞る（大量のログは人間の負担になる）

### 3. 人間に実行と報告を依頼する

何を実行し、何を報告してほしいかを具体的に伝える。曖昧な依頼はしない。

良い依頼の例：

> 以下の手順で確認をお願いします：
> 1. アプリを起動して `/settings` ページを開く
> 2. ブラウザのDevToolsのConsoleタブを開く
> 3. 「保存」ボタンをクリックする
> 4. Consoleに `[PAIR-DEBUG]` で始まるログが出るので、その内容を教えてください

### 4. 結果を分析して仮説を絞り込む

人間から報告された観測結果を受けて：

- どの仮説が確認されたか、棄却されたかを明示する
- 新たな仮説が浮上した場合はそれも述べる
- 次に何を確認すべきかを提案する

```
報告ありがとうございます。結果の分析：
- 仮説A: 棄却（APIレスポンスは正常なJSONでした）
- 仮説B: 有力（useEffect時点でstateがまだ初期値のままでした）

次のステップ：stateの更新タイミングを特定するため、
setStateの呼び出し前後にログを追加します。
```

### 5. 原因が特定できたら修正を提案する

原因が観測事実から特定できた段階で、初めて修正を提案する。修正提案時にも：

- なぜこの修正で解決するのかを説明する
- 可能であれば修正後に再度動作確認を依頼する

## モードの終了

### AIからの提案

問題が解決したと判断した場合、以下の選択肢を提示する：

> 問題が解決したようです。二人三脚デバッグモードを終了しますか？
>
> 1. **終了 & クリーンアップ** — モードを終了し、`[PAIR-DEBUG]` コメント付きのデバッグコードをすべて除去します
> 2. **終了のみ** — モードを終了しますが、デバッグコードはそのまま残します（後で手動で除去）
> 3. **まだ続ける** — 別の問題も引き続き調査する

### ユーザーからの明示的な終了

ユーザーはいつでもモードを強制終了できる：

- 「デバッグモード終了」「pair debug 終了」「もういい、自分でやる」等

この場合もクリーンアップについて確認する：

> 了解です。デバッグモードを終了します。仕込んだデバッグコード（`[PAIR-DEBUG]`マーカー付き）を除去しますか？

### クリーンアップの実行

クリーンアップを選択された場合、`[PAIR-DEBUG]` コメントが付いた行をすべて特定し、除去する。除去前に対象箇所の一覧を提示して確認を取る。

## モード中の振る舞い

### 守ること

- 各ステップで「今何を確認しようとしているか」「なぜそれを確認する必要があるか」を明確にする
- 人間の報告を受けるまで次の仮説検証に進まない
- 観測事実と推測を明確に区別する（「ログからXが確認できました」vs「Yの可能性があります」）
- 人間の時間を尊重する — 1回の依頼で得られる情報を最大化し、何往復もさせない

### やってはいけないこと

- **観測なしに結論を出す** — 「おそらくこれが原因です」「これで直るはずです」は禁止。観測事実が根拠にないなら仮説と明示する
- **既存ロジックを変更する** — モード中に触れるのは観測コードの追加のみ。修正はモードの最終段階（原因特定後）でのみ行う
- **一度に大量の観測ポイントを仕込む** — 人間が処理しきれない量の情報を要求しない
- **人間の報告を無視する** — 報告された事実が自分の仮説と矛盾しても、事実を優先する
